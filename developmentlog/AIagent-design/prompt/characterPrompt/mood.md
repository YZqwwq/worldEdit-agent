# 人格层 / 状态层（Mood Layer）

## 目标

人格层回答的问题是：**这个角色在当前阶段、当前会话里，呈现出怎样的行为状态**。

它不是“角色是谁”，而是“这个角色此刻以什么温度、主动性、解释密度、关系姿态来行动”。  
这层应当是可运行时变化的，可以受用户偏好、近期互动、任务场景影响。

## 这一层应该写什么

- 当前互动基调：温和、专注、克制、鼓励、严谨等
- 主动性：更主动承接，还是更强调确认
- 解释密度：简洁、平衡、详细
- 关系姿态：陪伴式、协作式、执行式中的偏向
- 当前阶段的行为倾向：先给结论、先澄清、先安抚、先行动
- 对近期上下文的承接偏好：更强地延续已确认对象、减少重复追问

## 这一层不应该写什么

- 不重写角色身份
- 不写具体 HTML 标签、富文本白名单、输出格式细则
- 不写工具名、节点名、数据库字段名
- 不把 mood 写成戏剧化的“情绪表演”
- 不让 mood 主导事实判断

人格层是**行为倾向层**，不是文学表演层。

## 推荐写法

人格层要写成“当前行为状态说明”，而不是散文。

推荐结构：

1. 当前互动温度
2. 当前主动性
3. 当前解释密度
4. 当前关系姿态
5. 当前阶段的优先行动习惯

## 示例骨架

```md
当前行为状态：

- 你此刻的互动温度应保持稳定、耐心、可信，不要显得兴奋过度，也不要像流程机器人。
- 你默认以“协作式承接”与用户交流：能承接的上下文尽量承接，能直接推进的步骤尽量推进。
- 解释密度保持中等偏低：先给有用结果，再按需要补充说明，不要一上来铺满规则或背景。
- 若上下文已经足够明确，应直接继续处理，不要重复要求用户再次提供世界观名称、人物名称或已确认对象。
- 若存在不确定性，应简洁指出不确定点，并提出最小必要追问，不要把问题扩展成冗长问卷。
```

## 可参数化方向

这层适合承接运行时人格状态，例如：

- `autonomy`
- `verbosity`
- `formality`
- `risk`
- 近期行为叙事摘要

但这些参数不应直接裸露给最终模型作为一堆数字；更适合先转成自然语言行为说明。

例如：

- 高 autonomy：更主动承接，减少“要不要我继续”
- 低 autonomy：重要分支前先确认
- 高 verbosity：适度补上下文与解释
- 低 verbosity：优先短句和结论

## 顺序与权重建议

人格层应该放在身份层之后、工具规则之前。

原因：

- 它需要继承身份层，不应盖过身份层
- 它需要在模型做工具决策前生效，影响“怎么理解和推进”
- 它不应放在最后，否则会和输出约束层打架

推荐顺序：

1. `character.md`
2. `mood.md`
3. 任务/上下文/工具规则
4. `expression.md`

## 设计原则

一句话概括：**人格层负责让模型知道“我现在以什么状态与用户相处”，而不是“我最后一句话该怎么排版”。**
