# 表述层 / 输出契约层（Expression Layer）

## 目标

表述层回答的问题是：**这一次输出最终应该怎么说，哪些能说，哪些不能说**。

这层不是角色身份，也不是行为状态，而是最终对用户可见文本的表达契约。  
它的职责是约束落字方式，保证自然、合规、可读、可渲染。

## 这一层应该写什么

- 输出对用户的可见风格约束
- 禁止暴露的内部信息
- 富文本输出允许的结构
- 回答组织方式
- 失败时的表达方式
- 调用工具后的表达清洗规则

## 这一层不应该写什么

- 不重写角色是谁
- 不重写当前情绪人格
- 不写大量世界观事实
- 不写模型思维链
- 不写“如何决定是否调用工具”这类前置推理规则

表述层只负责**最终可见输出**。

## 推荐写法

表述层应该尽量短、硬、明确。  
它比身份层和人格层更像“契约”，但不要写成过长的操作手册，否则会压住人格感。

推荐结构：

1. 输出目标
2. 禁止项
3. 组织方式
4. 富文本约束
5. 失败兜底

## 示例骨架

```md
输出契约：

- 直接对用户说话，不要描述内部流程，不要汇报“我正在调用工具”“我将查询数据库”“当前任务状态如何”。
- 不要向用户暴露内部标识或内部结构，包括但不限于：entityId、worldId、taskId、executionId、notificationId、数据库字段名、节点名、工具名。
- 工具返回的结构化数据必须先转成自然语言，再呈现给用户；除非用户明确要求原始标识，否则不要展示原始字段。
- 输出优先给结论或有效结果，再补充必要说明；不要一上来写成流程报告。
- 如果需要使用富文本，只使用系统允许的安全子集，不要输出不受支持的标签、脚本或样式。
- 如果信息不足，需要最小必要追问；追问应简洁、单轮、聚焦，不要展开成多项清单。
- 如果执行失败或存在不确定性，应自然说明问题，并明确下一步建议，不要输出内部报错堆栈或系统术语。
```

## 顺序与权重建议

表述层应放在 system prompt 的较后位置，但不要承担人格塑造职责。

原因：

- 它越靠后，对最终措辞影响越强
- 但如果它同时承担“人格描述”，会把前面的身份层和人格层冲淡

因此这层要遵守一个原则：

- 只写**硬约束**
- 不写“你是一个怎样的人”

## 与后处理的关系

表述层很重要，但不能单独承担全部合规工作。  
对于以下高风险项，建议后续由输出后处理再做一次硬校验：

- 是否泄露内部 ID
- 是否直接复述工具结构化原文
- 富文本标签是否越界
- 是否出现内部流程措辞

也就是说：

- `expression.md` 负责软约束生成
- 输出 validator 负责硬兜底

## 设计原则

一句话概括：**表述层负责让模型知道“这次最终该怎么对用户说”，而不是“这个角色是谁”。**
